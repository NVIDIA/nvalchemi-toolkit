The problem, and the shape of the solution

One GPU hits two walls as systems grow:

  • Memory — a big system won't fit (≈44k atoms for MACE, ≈4k for UMA on an 80 GB H100).
  • Time — MD is thousands of sequential steps; one GPU is too slow.

Fix: give each GPU a slice of the atoms. More GPUs → bigger and faster.

The hard part isn't splitting — it's staying bit-for-bit correct while the atoms move every step.

One principle: the model writes zero parallel code. It declares what it produces; the framework owns how it splits. Two ways to split: by space (Halo) or by atom-list (Graph-parallel).

Strategy 1 — Halo: split by space (MACE, AIMNet2)

  1. Partition the box into regions — one GPU each.
  2. Borrow ghosts: a thin shell of border atoms from neighbors.
  3. Message-pass: ghosts let boundary atoms see real neighbors.
  4. Refresh between layers — deeper layers stay correct.
  5. Keep only your atoms' forces.
The shell is refreshed between layers, so it stays a constant thin width — never growing with depth. Small shell → low memory → fit much larger systems.

Strategy 2 — Graph-parallel: split the atom list (UMA)

  1. Split the atom list evenly — no geometry needed.
  2. Divide the edges so the compute is shared.
  3. All-gather each layer — every GPU briefly sees all atoms, so message passing is exact.
  4. Reduce forces back to each atom's owner.
No spatial shell → no geometry constraints. Ideal for dense systems, where a halo shell would cover nearly the whole box.

Bring your own — everything

The core design bet: every layer is a swappable, declared component — not hard-wired.

▸ your Model

Any PyTorch message-passing model — declare a small spec, no rewrite. (Examples: a plain MPNN, a graph transformer.)

▸ your Strategy

Halo and Graph-parallel ship built-in, behind one interface — add a new scheme without touching models or dynamics.

▸ your Hook

Attach custom per-step logic — monitoring, logging, control — running correctly across all GPUs.

▸ your Dynamics

Any integrator: NVE, NVT, NPT, NPH, FIRE. And compose models (potential + electrostatics) in a pipeline.

Works today: MACE · UMA · AIMNet2 · PME · Ewald · DFTD3 · LJ · eager + torch.compile (zero recompiles in MD) · and matches a single GPU to ~10⁻⁶.

Multi-node — and we win at scale

Both scale across nodes. Below ~40k atoms fairchem's leaner cross-node comm leads; at ≥40k atoms we overtake — 1.1–1.3× faster.

Same crossover shape as single-node, shifted right by the extra cross-node communication — the regime production runs live in.
Our curve is also steadier at the top end, scaling cleanly to 71,874 atoms across the 16 GPUs (~68 GB/GPU) — about double the single-node ceiling. More GPUs ⇒ proportionally larger systems.

Takeaways

  • One clean design distributes many models — science stays pure, parallelism lives in one place.
  • Bring-your-own model, strategy, hook, and dynamics — the whole stack is swappable.
  • Two strategies: Halo (by space) and Graph-parallel (by atom list); eager and compiled; NVT/NPT/NPH.
  • Results are bit-for-bit correct vs. a single GPU (~10⁻⁶).
  • Near-linear UMA speedups single-node, and it scales across nodes (16 GPUs, IB) — overtaking fairchem's own distributed MD at production scale (≥40k atoms).
